## Задание 2. FURPS+ таблица архитектурно-значимых требований

На основе анализа потребностей стейкхолдеров и бизнес-целей была создана детальная FURPS+ таблица, классифицирующая все требования к системе открытия депозитов и кредитов онлайн.

**Ключевые категории требований:**
- **Функциональные (F)**: Подача заявок через сайт и интернет-банк, персонализированные ставки, СМС-подтверждения
- **Удобство использования (U)**: Интуитивный интерфейс, корпоративный дизайн, адаптивность
- **Надежность (R)**: Доступность 99.9%, защита от перегрузки АБС, отказоустойчивость
- **Производительность (P)**: Отклик в миллисекундах, кэширование, горизонтальное масштабирование
- **Поддерживаемость (S)**: Использование знакомых технологий, модульная архитектура, документирование
- **Ограничения (+R)**: Шифрование данных, регуляторные требования, технические ограничения legacy-систем

## Задание 3. Концептуальная архитектура открытия депозитов (MVP)

Разработана концептуальная архитектура MVP для онлайн открытия депозитов в формате ADR (Architecture Decision Record) с использованием модели C4.

**Ключевые архитектурные решения:**
- **Deposits Service** - новый микросервис для управления депозитами с кэшированием ставок
- **API Gateway** - единая точка входа с аутентификацией и rate limiting
- **Event Bus (Kafka)** - асинхронная обработка уведомлений и аудита
- **ABS Adapter** - защитный слой для интеграции с АБС без перегрузки core-системы

**Технологический стек:** Java Spring Boot, PostgreSQL, Redis, Apache Kafka, Kong

## Задание 4. Передача ставок в кол-центр

В рамках подготовки к запуску MVP депозитной платформы выявлена потребность в обеспечении операторов кол-центра актуальными ставками по депозитам. Учитывая целевую аудиторию банка (пожилые клиенты), ожидается увеличение количества звонков с вопросами о новых процессах открытия депозитов.

**Бизнес-требования:**
- Операторы внутреннего кол-центра должны иметь доступ к актуальным ставкам для консультаций
- Партнерский кол-центр должен получать ставки в файловом формате (ограничения их IT-системы)
- Минимальные изменения в существующих системах кол-центра и АБС
- Обеспечение надежности и актуальности данных

**Архитектурное решение:**
- **Расширение Rates Service**: Добавление API endpoint'ов для кол-центра и механизма генерации файлов
- **SFTP File Server**: Новый компонент для безопасного обмена файлами с партнером
- **Dual Integration**: API для внутреннего кол-центра, SFTP для партнерского
- **Минимальные изменения**: АБС и системы кол-центров остаются без архитектурных модификаций

**Технологические решения:**
- REST API через существующий API Gateway для внутреннего кол-центра
- SFTP-протокол с автоматической генерацией CSV/JSON файлов для партнера
- Redis кэширование для быстрого отклика операторов
- Scheduler для регулярного обновления ставок и файлов
