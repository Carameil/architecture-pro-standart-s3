### **Решение**
Архитектурное решение основано на введении промежуточной сервисной шины между клиентскими каналами и АБС для защиты core-системы и обеспечения масштабируемости.

**Ключевые архитектурные решения:**

1. **Deposits Service** - новый микросервис для управления депозитами, который:
   - Получает заявки от веб-каналов
   - Кэширует актуальные ставки из АБС
   - Управляет жизненным циклом заявок
   - Интегрируется с АБС через controlled API

2. **API Gateway** - единая точка входа для всех внешних запросов:
   - Маршрутизация запросов к соответствующим сервисам
   - Аутентификация и авторизация
   - Rate limiting для защиты от перегрузки

3. **Event Bus (Kafka)** - для асинхронной обработки:
   - Отправка уведомлений
   - Аудит операций
   - Интеграция между сервисами

4. **Модификация существующих систем**:
   - Расширение Сайта для приема заявок
   - Добавление депозитного модуля в Интернет-банк
   - API-слой для АБС без изменения ядра

**Технологический стек:**
- Deposits Service: Java Spring Boot (совместимость с командой)
- База данных: PostgreSQL (для новых сервисов)
- Кэш: Redis (для ставок и сессий)
- Очереди: Apache Kafka
- API Gateway: Kong

### **Альтернативы**

**Альтернатива 1: Прямая интеграция с АБС**
- Прямое подключение веб-каналов к АБС без промежуточного слоя
- Преимущества: Простота реализации, меньше компонентов
- Недостатки: Риск перегрузки АБС, нарушение изоляции, сложность масштабирования

**Альтернатива 2: Монолитное расширение интернет-банка**
- Добавление функциональности депозитов напрямую в существующий ИБ
- Преимущества: Использование существующей инфраструктуры
- Недостатки: Увеличение сложности монолита, зависимость от подрядчика

**Альтернатива 3: Полная замена интернет-банка**
- Разработка нового интернет-банка с нуля на микросервисной архитектуре
- Преимущества: Современная архитектура, полная свобода в технологиях
- Недостатки: Высокие затраты, длительные сроки, высокие риски

**Недостатки, ограничения, риски**

**Недостатки выбранного решения:**
1. Увеличение сложности архитектуры за счет добавления новых компонентов
2. Необходимость разработки API-слоя для АБС
3. Потребность в экспертизе по Kafka и микросервисам

**Ограничения:**
1. АБС остается узким местом для операций, требующих консистентности
2. Ограниченные возможности модификации core-банковских процессов в рамках MVP
3. Зависимость от стабильности интеграций с legacy-системами

**Риски:**
1. **Технический риск**: Сложность интеграции с АБС через новый API-слой
2. **Операционный риск**: Необходимость обучения команды новым технологиям
3. **Производительный риск**: Возможные задержки из-за дополнительных слоев абстракции
4. **Безопасность**: Увеличение поверхности атак за счет новых компонентов

**Пути снижения рисков:**
- Поэтапное внедрение с тщательным тестированием
- Мониторинг производительности на каждом этапе
- Обучение команды и документирование всех процессов
- Применение security best practices для всех новых компонентов
