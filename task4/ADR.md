### **Нефункциональные требования**
Опишите здесь нефункциональные требования и архитектурно значимые требования.

|**№**|**Требование**|
| :-: | :- |
|1|**Производительность**: API для получения ставок должен отвечать не более чем за 500ms для оптимального пользовательского опыта операторов|
|2|**Надежность**: Система должна работать в режиме 24/7 с доступностью не менее 99.5%, включая резервирование SFTP-сервера|
|3|**Интеграция**: Минимальные изменения в существующих системах - АБС, система кол-центра остаются без архитектурных модификаций|
|4|**Актуальность данных**: Ставки должны обновляться автоматически не реже чем раз в день, с возможностью форсированного обновления|
|5|**Безопасность**: Передача файлов через защищенный SFTP-протокол, доступ к API только для авторизованных систем|
|6|**Поддерживаемость**: Использование существующих технологий банка (Java Spring Boot) и простых форматов файлов (CSV/JSON) для партнеров|
### **Решение**

**Архитектурное решение основано на расширении существующего Rates Service из депозитной платформы с добавлением новых интеграционных возможностей для кол-центров.**

**Ключевые архитектурные решения:**

1. **Расширение Rates Service**: Добавление API endpoint'ов для кол-центра и механизма генерации файлов для SFTP
2. **SFTP File Server**: Новый компонент для обмена файлами с партнерским кол-центром
3. **Минимальные изменения в legacy-системах**: АБС и система кол-центра остаются без изменений
4. **Кэширование ставок**: Redis используется для быстрого доступа к актуальным ставкам

**Технологические решения:**
- **API для внутреннего кол-центра**: REST API через существующий API Gateway с аутентификацией
- **Файловый обмен с партнером**: SFTP-протокол с автоматической генерацией CSV/JSON файлов  
- **Синхронизация**: Scheduler для регулярного обновления ставок из АБС
- **Безопасность**: HTTPS для API, SFTP с SSH-ключами для файлового обмена

**Логика принятия решений:**
- Переиспользование существующего Rates Service минимизирует время разработки и поддерживает консистентность
- SFTP выбран как наиболее универсальный протокол для B2B интеграций
- Кэширование обеспечивает быстрый отклик для операторов кол-центра
- Файловый формат позволяет партнеру работать без изменений в их системе

### **Альтернативы**

**Альтернатива 1: Прямая интеграция с АБС для каждого кол-центра**
- Описание: Каждый кол-центр получает прямой доступ к АБС для получения ставок
- Преимущества: Всегда актуальные данные без промежуточного кэширования
- Недостатки: Высокая нагрузка на АБС, сложность обеспечения безопасности, отсутствие изоляции

**Альтернатива 2: Email-рассылка со ставками**
- Описание: Автоматическая email-рассылка файлов со ставками операторам кол-центров
- Преимущества: Простота реализации, знакомый процесс для операторов
- Недостатки: Не автоматизировано, риск человеческих ошибок, низкая скорость обновления

**Альтернатива 3: Веб-портал для операторов кол-центра**
- Описание: Создание отдельного веб-интерфейса для просмотра ставок
- Преимущества: Богатая функциональность, удобный интерфейс
- Недостатки: Дополнительные затраты на разработку и поддержку, необходимость обучения операторов

**Недостатки, ограничения, риски**

**Недостатки выбранного решения:**
1. Зависимость от стабильности SFTP-сервера для партнерского кол-центра
2. Задержка обновления файлов может привести к работе с устаревшими ставками у партнеров
3. Необходимость мониторинга двух разных каналов передачи данных (API и SFTP)

**Ограничения:**
1. Файловый формат ограничивает гибкость передачи сложных данных партнерскому кол-центру
2. SFTP не поддерживает real-time уведомления об изменениях ставок
3. Партнерский кол-центр должен самостоятельно реализовать обработку получаемых файлов

**Риски:**
1. **Операционный риск**: Сбой SFTP-сервера может привести к работе партнеров с устаревшими ставками
2. **Интеграционный риск**: Изменения в формате данных АБС могут повлиять на оба канала передачи
3. **Безопасности**: Компрометация SFTP-доступа может привести к несанкционированному доступу к ставкам
4. **Производительности**: Увеличение количества запросов от кол-центра может повлиять на производительность Rates Service

**Митигации рисков:**
- Резервирование SFTP-сервера и мониторинг его доступности
- Версионирование API и обратная совместимость форматов файлов
- Использование SSH-ключей и регулярная ротация доступов для SFTP
- Кэширование и rate limiting для защиты Rates Service от перегрузки
